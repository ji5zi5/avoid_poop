# 실시간 멀티 생존전 + 보스 패턴 리프레시 설계

## 목표
현재 싱글 플레이 기반 브라우저 게임을 유지하면서, 보스 패턴을 더 읽기 쉽고 재미있게 정리하고, 로그인 기반의 8인 실시간 생존전 멀티플레이를 추가한다.

## 배경
현재 프로젝트는 `React + Canvas` 프론트엔드와 `Fastify` 백엔드, `shared` 계약 패키지로 구성되어 있다. 싱글 플레이는 브라우저 내부 게임 엔진(`frontend/src/game/`)이 담당하고, 서버는 로그인/기록 저장만 처리한다. 멀티플레이를 넣으려면 실시간 상태 동기화와 판정 권한이 서버 쪽으로 이동해야 한다.

## 사용자 승인된 요구사항
### 싱글
- 재미없는 보스 패턴은 삭제한다.
- 새 패턴을 추가한다.
- 일반/하드 체감 차이를 더 크게 만든다.
- 패턴명은 플레이 화면에 노출하지 않는다.

### 멀티
- 메인 메뉴를 `싱글 / 멀티`로 분리한다.
- 멀티는 실시간 온라인 플레이어 간 생존전이다.
- 최대 8명까지 같은 맵에서 서로 보인다.
- 마지막 1명이 승리한다.
- 죽으면 관전 모드로 남는다.
- 로그인한 사용자만 입장할 수 있다.
- 기본 진입은 `방 코드 입장`, 추가 진입은 `빠른 입장`이다.
- 방장은 `시작`, 다른 사람은 `준비`를 누른다.
- `길막`은 방 옵션으로 켜고 끌 수 있다.
- 디버프는 기본 강도 2, 커스텀 방에서 3까지 허용한다.
- 디버프는 `디버프 전용 아이템`을 먹으면 서버가 `랜덤 1명`에게 적용한다.
- 멀티도 싱글처럼 `라운드 + 보스` 구조를 유지한다.
- 연결이 끊기면 10초 재접속 유예를 둔다.

## 비목표
- 글로벌 랭크 매치 시즌 시스템
- 음성 채팅/텍스트 채팅
- 치트 방지 완전 보장
- 100명 이상 대규모 룸
- 모바일 앱 패키징

## 접근 방식
### 선택안
싱글 엔진은 계속 클라이언트 중심으로 유지하고, 멀티는 별도의 `서버 권한(server-authoritative)` 실시간 엔진을 추가한다.

### 이유
- 싱글을 다 뜯어고칠 필요가 없다.
- 멀티에서 위치, 충돌, 탈락, 아이템, 보스 패턴을 서버 기준으로 맞춰야 판정이 안 꼬인다.
- 현재 백엔드는 이미 로그인/세션을 처리하므로, 인증된 WebSocket 연결만 추가하면 된다.

### 배제한 안
1. **클라이언트 판정 + 서버 중계**
   - 구현은 빠르지만 위치/충돌/디버프/재접속 판정이 쉽게 어긋난다.
2. **싱글/멀티 전부 서버 권한으로 전면 재구성**
   - 구조는 깔끔하지만 현재 프로젝트 기준 과하다.

## 시스템 설계
### 프론트엔드
기존 `App.tsx`의 화면 분기를 확장해서 멀티 홈, 로비, 멀티 인게임, 멀티 결과 화면을 추가한다.

예상 화면 흐름:
- 인증 화면
- 메인 홈: `싱글 / 멀티`
- 멀티 홈: `빠른 입장 / 방 만들기 / 방 코드 입장`
- 로비: 인원 목록, 준비 상태, 방 옵션, 시작 버튼
- 멀티 인게임: 같은 맵, 플레이어 닉네임/상태, 남은 인원, 라운드/보스 HUD
- 멀티 결과: 최종 순위, 승자, 다시하기/로비 복귀

싱글 화면은 유지하되, 보스 패턴 세트와 난이도 계수만 조정한다.

### 백엔드
기존 Fastify 앱에 멀티 모듈을 추가한다.

예상 백엔드 모듈:
- `server/src/modules/multiplayer/http.routes.ts`
  - 방 생성, 방 코드 입장, 빠른 입장, 로비 정보 조회
- `server/src/modules/multiplayer/socket.gateway.ts`
  - WebSocket 연결, 세션 검증, 룸 브로드캐스트
- `server/src/modules/multiplayer/matchmaking.service.ts`
  - 빠른 입장 큐, 대기 중 룸 배정
- `server/src/modules/multiplayer/room.service.ts`
  - 룸 상태, 준비, 방장 시작, 방 옵션, 관전, 재접속
- `server/src/modules/multiplayer/game.service.ts`
  - 서버 권한 게임 tick, 플레이어 상태, 탈락, 승리 판정
- `server/src/modules/multiplayer/debuff.service.ts`
  - 랜덤 대상 선정, 디버프 지속시간/강도 관리
- `server/src/modules/multiplayer/result.service.ts`
  - 경기 결과 저장

## 실시간 규칙
### 서버 권한 범위
서버가 아래를 결정한다.
- 플레이어 실제 위치/속도
- 낙하물 스폰/이동
- 아이템 스폰/획득
- 디버프 대상/적용
- 플레이어 충돌(길막 ON일 때)
- 탈락/관전 전환
- 라운드/보스 전환
- 승자 판정

클라이언트는 입력 전송과 보간 렌더링만 담당한다.

### 룸 규칙
- 인원 최대 8명
- 로그인 세션 필수
- 방장만 시작 가능
- 참가자는 준비 토글 가능
- 방 옵션
  - `bodyBlock`: 길막 ON/OFF
  - `debuffTier`: 2 또는 3

### 죽음/관전
- 피격 또는 조건 충족으로 탈락하면 조작을 잃는다.
- 관전자로 전환되어 경기 종료까지 룸에 남는다.
- 마지막 생존자 1명이 승리한다.

### 재접속
- 연결 종료 시 10초 유예
- 유예 안에 동일 계정으로 복귀하면 원래 슬롯 복귀
- 시간 초과 시 탈락 처리

## 멀티 전용 아이템/디버프
### 디버프 아이템
멀티에서만 떨어지는 전용 아이템으로 둔다. 일반 회복/무적 계열과 분리해서 읽기 쉽게 만든다.

### 기본 디버프(기본 강도 2)
- 슬로우
- 좌우 반전
- 짧은 입력 지연

### 커스텀 방 추가 허용(강도 3)
- 시야 방해 강화
- 아이템 봉인 짧게

대상은 서버가 생존자 중 랜덤 1명을 골라 적용한다.

## 싱글 패턴 리프레시 방향
멀티 이전에 싱글 보스 패턴 세트를 다듬는다.

- 삭제 후보: 반복 체감이 강하고 차별점이 약한 패턴
- 추가 방향: 대왕똥, 통로형, 페이크형 중 읽기 쉬운 패턴 우선
- 일반 모드: 읽히는 패턴 위주, 더 긴 전조/낮은 속도/적은 연결 수
- 하드 모드: 패턴 풀 확대, 더 짧은 전조/빠른 속도/더 많은 연결 수
- 패턴명 UI는 계속 숨긴다.

## 계약 및 데이터 모델
### shared 계약 확장
새 계약 파일 추가를 권장한다.
- `shared/src/contracts/multiplayer.ts`

예상 타입:
- `LobbyRoomSummary`
- `LobbyPlayer`
- `RoomOptions`
- `CreateRoomPayload`
- `JoinRoomPayload`
- `QuickJoinPayload`
- `SocketClientEvent`
- `SocketServerEvent`
- `MultiplayerResultEntry`

### DB 저장
현재 DB는 `users`, `records` 중심이다. 멀티 추가 시 최소한 아래가 필요하다.
- `multiplayer_matches`
- `multiplayer_participants`

1차에서는 룸 상태는 메모리로 두고, 종료 결과만 DB에 기록한다.

## 기술 제약
- 현재 백엔드는 Fastify만 사용 중이며, 실시간 WebSocket 처리를 위해 추가 수단이 필요하다.
- 저장소 규칙상 새 의존성은 명시적 승인 없이 바로 추가하지 않는다.
- 따라서 구현 시작 전 `@fastify/websocket` 또는 `ws` 도입 여부를 명시적으로 확정해야 한다.
- 만약 의존성 승인이 지연되면, 문서/계약/로비 HTTP 흐름과 싱글 패턴 리프레시부터 먼저 진행한다.

## 테스트 전략
### 싱글
- 보스 패턴 큐/난이도/공정성 테스트 유지
- 일반/하드 차이 테스트 추가
- 삭제/추가된 패턴에 대한 회귀 테스트 추가

### 멀티 서버
- 방 생성/입장/준비/시작 테스트
- 빠른 입장 매칭 테스트
- 재접속 10초 유예 테스트
- 디버프 대상 랜덤 선정 테스트
- 마지막 1명 승리 테스트

### 멀티 클라이언트
- 화면 분기 테스트
- 로비 상태 UI 테스트
- 관전 전환 UI 테스트

### 통합
- 로그인 후 룸 생성 → 준비 → 시작 → 결과 저장 시나리오 검증

## 리스크
1. **실시간 동기화 복잡도**
   - 완화: 서버 권한 단일 tick 유지, 클라이언트는 보간만 담당
2. **현재 싱글 엔진과 멀티 엔진의 중복**
   - 완화: 패턴 정의/렌더 자산은 공유, 판정 루프만 분리
3. **의존성 승인 이슈**
   - 완화: 승인 전까지는 싱글 패턴 리프레시 + shared 계약 + 로비/HTTP 작업 우선
4. **8인에서 길막 옵션 시 충돌 복잡도 증가**
   - 완화: 길막 ON은 1차에 단순 원형/축 정렬 충돌만 적용

## 권장 구현 순서
1. 싱글 패턴 리프레시
2. shared 멀티 계약
3. 멀티 홈/로비 UI
4. 룸/매치메이킹 HTTP
5. WebSocket 게이트웨이
6. 서버 권한 멀티 엔진
7. 관전/재접속
8. 결과 저장
9. 검증 및 밸런스 조정
